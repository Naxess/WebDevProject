Django - Forms

Django's form functionality can be used to simplify and automate the process of handling forms and can do it more securely. Django handles three distinct parts of the work involved in forms:

  • Preparing and restructuring data to make it ready for rendering
  • Creating HTML forms for the data
  • Receiving and processing submitted forms and the data from the client

While it's certainly possible to write the code manually, Django already has this functionality built-in so it's more time and labor efficient to make use of it.

HTML Forms

In HTML, a form is a collection of elements inside <form> tag that allow users to do things like entering text, selecting options, manipulating objects/controls, etc.. and then send that information back to the server. Some of these elements are built-in and self-contained, while others may require CSS or Javascript to function. Like <input> elements, forms must specify two things:

  • where. This is the URL to which the data corresponding to the user's input should be returned.
  • how. This is the HTTP method the data should be returned by.

As an example, the login form for the Django admin contains three <input> elements. The input for the username has the type attribute set to "text", the password input has the type attribute set to "password", and the submit button has the type attribute set to "submit". it also tells the browser that the form data should be sent to the URL specified in the <form>'s action attribute - "/admin/" - and that it should be sent using the HTTP mechanism specified by the method attribute: "post". When the submit button is pressed, the data is sent to /admin/.

GET and POST

GET and POST are the only HTTP methods to use when dealing with forms.

When a form is returned via the POST method, the browser bundles up the form data, encodes it for transmission, send it to the server, and then receives back its response.

The GET method, by contrast, bundles the submitted data into a string, and uses this to compose a URL. The URL contains the address where the data must be sent, as well as the data keys and values.

GET and POST are typically used for different purposes. Any request that could be used to change the state of the system (e.g. changes to the database) should use POST. GET should be used only for requests that do not affect the state of the system.

GET is also unsuitable for a password form, because the password would appear in the URL, and thus, also in the browser history and server logs, all in plain text. It's also unsuitable for large quantities of data, or for binary data, such as an image. A web application that uses GET requests for admin forms is a security risk: it can be easy for an attacker to mimic a form's request to gain access to sensitive parts of the system.
POST, coupled with other protections like Django's CSRF protection offers more control over access. GET is more suitable for things like a web search form, because the URLs that represent a GET request can easily be bookmarked, shared, or resubmitted.

The Form Class

Central to this system of functionality is the Form class. This class describes a form and determines how it works and how it appears. The fields in a form class map to HTML form <input> elements. These fields are classes as well; they manage form data and perform validation when a form is submitted. (e.g. Datefield, FileField, CharField, etc..). A form field is represented to the user in the browser as an HTML widget - a piece of UI machinery. Each field type has a corresponding Widget class, but can be overriden as needed.

Instantiating, Processing, and Rendering Forms

Rendering an object in Django typically follows this schema:

  • Retrieve the object into the views.py class
  • Pass the object into the template context
  • Expand it to HTML markup using template variables

The way a form is rendered is very similar to the way other objects but there are some differences:

  • If a model instance contains no data, there's little that can be done with it, whereas an unpopulated form is typically rendered without any value.
  • When handling a model instance in the view, it's typically retrieved from the database. When dealing with a form, it's typically instantiated in the view.
  • When instantiating a form, it can be left empty or prepopulated with data from the following sources:
    • Data from a saved model instance
    • Data that has been collated from other sources
    • Data received from a previous HTML form submission

Building a Form
An example of a manually-created form:
{% verbatim example_manual_form %}
    <form action="/your-name/" method="post">
        <label for="your_name">Your name: </label>
        <input id="your_name" type="text" name="your_name" value="{{ current_name }}">
        <input type="submit" value="OK">
    </form>
{% endverbatim example_manual_form %}

The above example tells the browser to return the form data to the URL: /your-name/, using the POST method. It will display a text field, labeled "Your name:", and a button marked "OK". If the template context contains a current_name variable, that will be used to pre-fill the your_name field. The view that renders this template containing the HTML form needs to supply the value for the current_name variable.

When the form is submitted, the POST request that is sent to the server will contain the form data. It needs a view corresponding to the /your-name/ URL, which will find the appropriate key-value pairs in the request and then process them.

This is a simple form but fairly time-consuming. Some more advanced features that could be added are browser-side validation and more complex fields (e.g. datepicker). In the below sections, we'll demonstrate a faster and easier way to accomplish the same task using Django's form creation feature.

Creating the form class

The first step to using the Forms feature is to create a Python file named forms.py in the root directory of the desired application. Then, in the forms.py file, import forms from the django module: 

    from django import forms

The next step is to define a Form class. Create a class with your desired name that inherits from the forms.Form class. Syntax: 

    from django import forms
    class ExNameForm(forms.Form):

Finally, we need to add input fields to the class we just created; these fields should correspond to an HTML input element type (e.g. an input element with type="text" should use a CharField, type="email" should use EmailField, etc..). 

    from django import forms
    class ExNameForm(forms.Form):
        your_name = forms.CharField(label="Enter your name:", max_length=100)

In the CharField, the label parameter fulfills the same role as the <label> element, providing a description for the label. The max_length parameter specifies the upper limit for the number of characters permitted for the input. There are other optional CharField parameters: min_length, strip, and empty_value.

Form instances also have an is_valid() method, which runs validation routines for all its fields. When this method is called, if all fields contain valid data, it will do two things: return True and place the form's data in its cleaned_data attribute.

The form, when rendered, will be the equivalent of the following HTML (minus the <form> tags, we'll have to supply those ourselves): 

    <label for="your_name">Enter your name: </label>
    <input id="your_name" type="text" name="your_name" maxlength="100" required>

Editing the views.py

The form data of a webpage sent back to a Django website will be processed by a corresponding view, typically the same view that rendered the template for that webpage. This permits reuse of some of the same logic.

To handle the form, we need to instantiate it in the view for the URL where we want it to be published: 

    from django.http import HttpResponseRedirect
    from django.shortcuts import render

    from ExampleApp.forms import ExNameForm # The form needs to be imported before it can be used

    def get_name(request):
        if request.method == "POST": # Processes the form data if it's a POST request
            name_form = NameForm(request.POST) # Creates a form instance and populates its with data from the request
            if form.is_valid(): # Checks whether the form instance's fields are valid
                ...
                # In the code body, we can process the data using form.cleaned_data if needed
                form_data = name_form.cleaned_data
                name = form_data['your_name']
                ...
                return HttpResponseRedirect("/thanks/") # Redirects to a new URL when processing is finished
        else: # If its a GET request, then we return an unpopulated form
            ex_name_form = ExNameForm()
        return render(request, "name.html", {"exnameform": ex_name_form}

Editing the template

To add the form to the HTML page, add a csrf_token, then create a template tag, supplying it with the name of the desired form. Create an <input> element tag with a type attribute set to "submit". Enclose the above elements with <form> element tags, setting its action attribute to the desired URL and method attribute to "post". See example below: 

{% verbatim template_example %}
    <form action="/first_app/your-name/" method="post">
        {% csrf_token %}
        {{ exnameform }}
        <input type="submit" value="Submit">
    </form>
{% endverbatim template_example %}

All of the example form's fields and their attributes will be unpacked into HTML markup from that {% verbatim exnameform %}{{ exnameform }}{% endverbatim exnameform %} by Django's template language. Optionally, the template variable can also call the as_p method, which will allow the form to be presented in the format of a <p> tag. If used, it should improve the layout of the form (preferences may vary, obviously). Example:
{% verbatim exnameform %}{{ exnameform.as_p }}{% endverbatim exnameform %}

The {% verbatim csrf_token %}{% csrf_token %}{% endverbatim csrf_token %} that we added is used for protection against CSRF (Cross Site Request Forgeries). This is a type of attack that occurs when a malicious website contains a link, a form button, or some JavaScript that is intended to perform some action on your website, using the credentials of a logged-in user who visits the malicious site in their browser.

Editing the urls.py

Example urls.py:

    from first_app import views
    from django.urls import path
    urlpatterns = [
        path('your-name', views.get_name, name='your_name_form'),
        path('thanks', views.thanks, name='thanks'),
    ]

More information about Django Form classes

All form classes are created as subclasses of either django.forms.Form or django.forms.ModelForm. (ModelForm will be discussed in a different chapter.)

Bound vs Unbound form instances
  • A unbound form has no data associated with it. When displayed to the user, it will be empty or contain default values.
  • A bound form has submitted data, and hence can be used to tell if that data is valid. If an invalid bound form is rendered, it can include inline error messages telling the user what data to correct.

The form's is_bound attribute can used to ascertain whether a form has data bound to it.

Form Fields

The forms library comes with form fields other than CharField, such as BooleanField, DateField, ImageField, etc.. These fields utilize a widget, that, if not specified explicitly will use a default one. For each field, we can enable or disable settings that force an input, or edit the label/help text/error messages, as well as whether or not to enable/disable the field itself by editing its arguments (e.g. required, label, help_text, error_messages, disabled, etc..). There are also field-specific arguments that edit things such as max/min length, regex pattern matching, stripping, etc.. These field-specific ones may be individual to a particular field or to a type of field.

Core field arguments describe arguments that are common to all form fields. See below table for the full list:

Argument Description
required

Forces the user to supply an input. Attempting to pass in a None or "" (empty string) will raise a ValidationError. 

f = forms.CharField(required=False)
label

Creates a text description that will be displayed next to the form field's input, functions the same as how the <label> works. 

url = forms.URLField(label="Your website", required=False)
label_suffix

A string that is appended after the label is rendered, typically a semicolon ( : ). Can be customized to display another character. 

captcha_answer = forms.IntegerField(label="2 + 2", label_suffix=" =")
initial

This argument allows us to specify a default value for the input field. 

url = forms.URLField(initial="http://")
widget

This argument allows us to specify the desired widget to use when rendering the field.

help_text

This argument allows us to specify the descriptive text for the field. If this value is provided, the text will be displayed next to the field when it is rendered. 

sender = forms.EmailField(help_text="A valid email address, please.")
error_messages

This argument allows overriding the default messages that the field will raise. To use it, we need to pass in a dictionary with keys matching the error messages to be overriden. Example usage: 

name = forms.CharField(error_messages={"required": "Please enter your name"})
validators

This argument allows us to provide a list of validation functions for this field.

localize

This argument enables the localization of form data input, as well as the rendered output.

disabled

If this argument is set to True, disables the form field using the disabled HTML attribute. Prevents the user from editing the field.

template_name

This agrument allows us to specify a custom template to be used when the field is rendered with as_field_group(). By default, this value is set to "django/forms/field.html". Can be changed per field by overriding this attribute or more generally by overriding the default template.

Creating Custom Form Fields

If the built-in Field classes dont meet your needs, you can create custom Field classes. To do this, create a subclass of django.forms.Field. Its only requirements are that it implement a clean() method and that its __init__() method accept the core arguments mentioned above (required, label, initial, widget, help_text).

You can also customize how a field will be accessed by overriding get_bound_field().

Widgets

As seen in the Form Field Reference section, each field has a corresponding Widget class, which in turn corresponds to an HTML form widget such as <input type="text"> in the HTML. In most cases, the field will have a sensible default widget. For example, by default, a CharField will have a TextInput widget, that produces a <input type="text"> in the HTML. But if we needed <textarea> instead, we would need to specify the appropriate widget when defining the form field.

Field Data

Whatever data is submitted with a form, once it has been validated by calling is_valid() and if it returns True, the validated form data will be accessible from the form.cleaned_data dictionary. Technically the invalidated data is accessible via request.POST but it is not recommended to use the data in this state.

The data stored in the form.cleaned_data dictionary will have been converted into Python types for ease of use. Fields such as IntegerField and FloatField will be converted into Python int and float, respectively.

Some field types need to be handled differently - such as files. For uploaded files, we need to access them via request.FILES instead.
