Git - Advanced

How to set up a repository:

A Git repository is a virtual storage of your project. It allows developers to save versions of the code for later access.

Initializing a new repository: git init

First navigate to the desired directory, then execute the git init command.

git init ‹project_directory›

Cloning an existing repository: git clone

If a project has already been set up with a (central) repository. We can use the git clone command to obtain a copy of the repository locally (local copy AKA working copy). This is typically a one-time operation. Once a local repository has been created, all version control operations are managed through this local repo. 

git clone ‹repo_url›

The git clone command is used to create a copy or clone of a remote repository. Repository URLs follow this template: git@HOSTNAME:USERNAME/REPONAME.git.

Example: git@bitbucket.org:mario/super-mario-bros.git

  • Hostname: bitbucket.org
  • Username: mario
  • Reponame: super-mario-bros

Saving changes to the repo: git add and git commit

After the repo has been cloned and initialized, we can then commit file version changes to it. After a file has been created. We need to add it to the list of files in the repository staging area using the git add command. 

git add ‹changed_file›

If we want to add a directory instead, we can substitute the file for a directory:

git add ‹directory›

Alternatively, if we want to add any and all changed and untracked files in the repo, we can use the --all flag instead of a specific file.

git add --all

The primary function of the git add command is to promote pending changes in the working directory to the git staging area.

After this it has been added to the staging area, we then need to commit the file so it will be added to the history so future updates to the file will be tracked. To do this we need to use the git commit command. 

git commit -m "‹your_message›"

Git commits capture the state of a project at that point in time - a snapshot. These snapshots are committed to the local directory rather than the central repo.

Repo-to-repo collaboration: git push

The git push command is used to update the remote repo with your local changes. It takes the commits you created and transfers it to the remote repo.

Bare vs cloned repos:

If your repository was created using the git clone command, then it is already configured for remote collaboration. It is already pointed at the Git URL that you cloned it from. This means that once you make changes to a file and commit them, you can git push those changed to the remote repo.

If your repo was created using the git init command, then there's no remote repo to push your changes to. What you can do is use a service like Bitbucket and create the repo there. This service will provide a Git URL that you can add to your local repo thus allowing you to push the changes to the repo (on Bitbucket). But before we can do this, some configuration changes need to be done. (see Configuration % setup section)

A 'bare' repository refers to a repo created using the git init or git clone commands with a --bare flag. This allows users to use this bare repo as a remote repo. The most common use for bare repos is to create a remote central Git repo.

git init --bare
git clone --bare

Configuration & setup: git config and git remote

Once the remote repo has been set up, we need to add this repo's URL to your local git config and set up an upstream branch for your local branches. We can do this using the git remote command: 

git remote add ‹remote_name› ‹report_repo_url›

This command will map the remote repo url to a ref in your local repo under the designated remote name. Once this has been mapped, you can push local branches to it using the git push command: 

git push -u ‹remote_name› ‹local_branch_name›

This command pushes the local repo branch under local_branch_bane to the remote repo at remote_name.

Define the author for all commits by using the --global flag:

git config --global user.name ‹name›

Define the author for the current local repo by using the --local flag or by not passing a config level option:

git config --local user.name ‹name›
git config user.name ‹name›

Define the author email for commits by the current user:

git config --local user.email ‹email›

Git add

This command adds a change in the working directory to the staging area. It tells Git that you want to include updates to a particular file in the next commit. However, git add doesn't really affect the repo in any significant way. Those changes are not actually recorded until you run git commit.

Git commit

This command captures a snapshot of the project's current staged changes.

Comparing changes: git diff

Diffing is a function that takes two input data sets and outputs the changes between them. git diff is a multi-use command that when executed will run a diff function on Git data sources. These data sources can be commits, branches, files, etc..

Git stash

The git stash command temporarily shelves changes you've made (can be both staged and unstaged) to your working copy so you can work on something else. It's still there so you can come back and re-apply it later on. It is handy if you need to quickly switch context and work on something else, but you're mid-way through a code change and aren't quite ready to commit. 

git stash

This default option doesn't include unstaged changes

To re-apply stashed changes, we can use the git stash pop command.

git stash pop

.gitignore

Git can be explicitly told to ignore files. Using wildcards, we can have Git ignore all files with certain extensions such as log files (*.log).

Inspecting a Repository

Git status

Displays the state of the working directory and the staging area. It allows you to see which changes have been staged, which haven't, and which files aren't being tracked by Git.

Git tag

Tags are references to specific points in Git history. This command is gnerally used to capture a point in history that is used for a marked version release.

Git blame

Displays the author metadata attached to specific committed lines in a file. This can allows us to explore the history of specific code and answer questions about what, how, and why the code was added to a repo.

Git log

This command displays committed snapshots. It allows you to list the project history, filter it, and search for specific changes.

Undoing Changes

Git checkout

This command can be used to visit a particular commit. Every commit has a unique SHA-1 identifying hash. It is an easy way to load any saved snapshots onto your development timeline.

Git revert

This command functions like an "undo" command for tracked files. However, it is not a traditional "undo" operation in that instead of removing the commit from the project history, it figures out how to invert the changes introduced by the commit and appends a new commit with the resulting inverse content. Basically, it is used for undoing changes to a repo's commit history. git checkout and git reset are also "undo" commands but they move the HEAD and branch ref points to a specified commit. While git revert also takes a specified commit, it does not move the ref points to this commit. It will take the specified commit, inverse the changes from that commit, and create a new "revert commit". The ref pointers are then updated to point at the new revert commit, making it the tip of the branch.

Revert has 2 advantages over resetting:

  • It doesn't change the project history, which makes it a relatively safe operation for commits that have already been published to a shared repo.
  • Second is that it is able to target an individual commit at an arbirary point in the history whereas git reset can only work backward from the current commit. For example, if you want to undo an old commit with git reset, you would have to remove ALL of the commits that occurred after the target commit, remove it, then re-commit all of the subsequent commits.

Git reset

This command reverts to a previous state of a project by removing ALL subsequent commits from the specified commit. This is different from a git revert which undoes a single commit.

Git clean

This command functions like an "undo" command but operates on untracked files. Untracked files being files that have been created within your repo's working directory but have not yet been added to the repo's tracking index using the git add command.

Git rm

This command is used to remove individual files or a collection of files. The primary function of git rm is to remove tracked files from the Git index. It can also be used to remove files from both the staging index and working directory. There is no option to remove from only the working directory. The files being operated on must be identical to the files in the current HEAD. If there is a discrepancy between the HEAD version of a file and the staging index or working tree version, Git will block the removal. This is a safety mechanism to prevent removal of in-progress changes. Does not remove branches.

Rewriting History

Git commit --amend

This command allows the user to modify the most recent commit. It let you combine staged changes with the previous commit instead of creating an entirely new commit. It can also be used to simply edit the previous commit message without changing its snapshot. Amending does not just alter the most recent commit, it replaces it entirely, meaning the amended commit will be a new entity with its own ref. To Git, it will look like a brand new commit.

These commands allows you to edit the previous commit's message without altering its snapshot. The first one will open an editor while second one allows you to add a message in-line: 

git commit --amend
git commit --amend -m "‹new_message›"

Adding the --no-edit flag to the command allows you to amend the previous commit without changing the message: 

git commit --amend --no-edit

Example usage of --amend:

git add file1of2.py
git commit
# Forgot to add file2of2.py!
git add file2of2.py
git commit --amend -m "Forgot to add file2of2.py. Added it into the latest commit."

In short, this command allows you to take the most recent commit and add new staged changes to it. You can add OR remove changes from the Git staging area to apply with a --amend commit. If there are no changes staged, then it will prompt you to edit the last commit message.

Git rebase

Refers to the process of moving or combining a sequence of commits to a new base commit. From a content perspective, it is changing the base of your branch from one commit to another, making it appear as if you created your branch from a different commit. Git accomplishes this by creating new commits and applying them to the specified base.

Git rebase -i

Adding the -i flag allows you to run rebase interactive. This gives you the opportunity to alter individual commits in the process, rather than moving all commits. It gives a finer grained control over history modifications than a standard git rebase.

Git reflog

Reference logs, or "reflogs" are a mechanism Git uses to record updates applied to tips of branches and other commit references. Reflog allows you go back to commits even though they are not referenced by any branch or tag. After rewriting history, the reflog contains information about the old state of branches and allows you to go back to that state if necessary.

Git Terminology
Term Description
Repository ("Repo") Can be thought of as a directory that stores all of the files, folders, and content needed for your project. What it actually is, is the object database of the project, storing everything from the files themselves, to the version of thsoe files, comits, deletions, et cetera. Repos are not limited by the user, and can be shared and copied.
Remote A copy of the original branch. When cloning a branch, that new branch is a remote, or clone. Remotes can talk to the origin branch, as well as other rmeotes for the repo, to making communication between working branches easiers.
Branch A version of the repository that diverges from the main working project. Can be a new version of a repo, experimental changes, or personal forks of a repo for users to alter and test changes.
Git Checkout Command use to switch branches in a repo.
Master Refers to the primary branch of all repos. All committed and accepted changes should be on the master branch. You can work directly from the master branch, or create other branches.
Origin The conventional name for the primary version of a repo. It is the alias to the remote url. Git also uses "origin" as a system alias for pushing and fetching data to and from the primary branch. For example, git push origin master, when run on remote, will push the changes to the master branch of the primary repo database.
Main Refers to a local branch.
HEAD Refers to the currently checked-out branch's latest commit. Represents the default branch on the remote.
Detached HEAD Refers to when the HEAD does not point to any branch, but instead points to a specific commit or the remote repository.
Index Refers to the working, or staging, area of Git. Files that have been cahnged, added, and deleted will be staged within the index until you are ready to commit the files. To see what is set in your Git index, run git status within your repo. The green files are staged and ready to commit, whereas the red files have not yet been addedto staging for the next commit.
Fork Refers to the action of creating a copy of a repo.
Rebase Refers to integrating changes from one branch onto another. The difference between this and git merge is that git rebase creates a new set of commits applied on top of the largest branch, while git merge creates a new merge commit that combines the changes from both branches.
Fetch The act of downloading and copying that branch's files to your workstation. Multiple branches can be fetched at once and you can rename the branches whenr unning the command to suit your needs.
Pull/Pull Request If someone has changed code on a separate branch of a project and wants it to be review to add to the master branch, that someone needs to submit a pull request. These are request to ask the repo owners/maintainers to review the commits made, and then, if accepted, the commits are merged upstream (usually into the master branch). A pull happens when adding the changes to the master branch.
Push Updates a remote branch with the commits made to the current branch. You are "pushing" your changes onto the remote.
Cherry-picking Refers to taking an older commit, and rerunning it at a defined location. Git copies the changes from the original commit, then adds them to the current location. It is a command that enables arbitrary commits to be picked by reference and appended to the current working HEAD. It is the act of picking a commit from a branch and apply it to another.
Stash Refers to the act of setting aside the current changes for later use.
Tag Used to define which portions of a project's Git history is the most important. Typically used to note point releases of a project. Tags can be added to the commit while it is still being worked on or added after it has been pushed.

Common Git Commands
Command Explanation
git init [directory_path] Initializes an existing directory as a Git repository.
git clone [repo_url] Creates a clone of the specified repo onto the local machine. The repo can be remote or local.
git config user.name name Defines the author name to be used for all commits in the current repo.
git add directory or file Stage all changes in the directory or file for the next commit.
git commit -m "message" Commits the staged snapshot.
git status Lists which files are stages, unstaged, and untracked.
git log Displays the entire commit history.
git diff Shows the unstaged changes between your index and the working directory.

Undoing Changes
Command Explanation
git revert commit Creates a new commit that undoes all of the changes made in the selected commit, then applies it to the current branch.
git reset file Removes the file from the staging area, but leaves the working directory unchanged. Unstages a file without overwriting any changes.
git clean -n Shows which files would be removed from the working directory.
git clean -f Removes the selected files from the working directory.

Rewriting Git History
Command Explanation
git commit --amend Replaces the last commit with the staged changes and last commit combined.
git rebase base Rebases the current branch onto the base.
git reflog Shows a log of changes to the local repo's HEAD. Flags: --relative-date (shows dates), --all (shows all refs)

Git Branches
Command Explanation
git branch Lists all of the branches in the repo.
git branch branch_name Creates a new branch with the given name.
git checkout branch_name Checks out the branch with the given name.
git checkout -b branch_name Creates and checks out a new branch with the given name.
git merge desired_branch Merges the desired branch into the current branch.

Remote Repositories
Command Explanation
git remote add name url Creates a new connection to a remote repo. After adding a remote, you can use the given name as a shortcut for url in the other commands.
git fetch remote Fetches all of the branches from the remote repo. This also downloads all of the required commits and files from the remote repo.
git fetch remote branch Fetches only the specific branch from the remote repo.
git pull remote Fetches the specified remote's copy of the current branch and immediately merges it into the local copy.
git push remote branch Pushes the branch to the remote repo, along with the commits and objects. Creates a name branch in the remote repo if it doesn't exist already.

Git Reset
Command Explanation
git reset Resets the staging area to match the most recent commit, but leaves the working directory unchanged.
git reset --hard Resets the staging area and working directory to match the most recent commit and overwrites all changes in the working directory.
git reset commit Moves the current branch tip backwards to the specified commit, resets the staging area to match, but leaves the working directory alone.
git reset --hard commit Moves the current branch tip backwards to the specified commit, then resets both the staging area and working directory to match. Deletes uncommitted changes and all commits after the specified commit.

Git Rebase
Command Explanation
git rebase -i base Interactive rebases the current branch on the specified base. Launches an editor to specify commands for how each commit will be transferred onto the new base.

Git Pull
Command Explanation
git pull --rebase remote Fetches the remote's copy of the current branch and rebases it into the local copy. Uses git rebase instead of merge to integrate the branches.

Git Push
Command Explanation
git push remote local_branch [--force] Uploads the local branch onto a remote repo.
git push remote --force Forces the local repo onto a remote repo. It forces the push even if it results in a non-fast-forward merge.
git push remote --all Pushes all of the local branches onto the specified remote.

Git Diff
Command Explanation
git diff HEAD Shows the differences between the working directory and the last commit.
git diff --cached Shows the differences between the staged changes and the last commit.

Some useful commands:

First command forces the remote repo to discard any uncommitted changes and is updated with changes from the local repo. Second command forces the local repo to discard any local changes since the last commit and updates itself to match the last commit.

Command Explanation
git push origin --force Forces the local repo onto a remote repo. It forces the push even if it results in a non-fast-forward merge.
git reset --hard HEAD
git pull 		Discards local changes, then overwrites the local repo with the remote repo.
